项目文档《fix/项目文档.txt》评审报告（基于当前仓库代码）
日期：2026-05-04

结论：文档存在明显问题，当前不建议直接按该文档继续做“2.0阶段”开发。

一、核心问题概览
1) 文档把“已修复”写成既成事实，但代码中大量关键点并未落地，存在“文档与实现不一致”。
2) 文档缺少“影响范围、验收标准、回归测试清单、改动顺序”，执行风险高。
3) 文档没有区分“短期止血”和“结构性重构”，会导致开发中途反复返工。

二、逐条对照发现的问题（按文档条目）
1. Domain Object Fix（Player/Room 不应是 Spring Bean）
- 文档声称应移除 @Component。
- 现状：Player、Room 仍保留 @Component。
- 影响：与“领域对象非Bean”的设计原则冲突，易造成生命周期与状态来源混乱。

2. Dependency Injection Fix（注入 RoomManager 而非 Room）
- 文档建议用 RoomManager 统一访问。
- 现状：DiscoveryService 仍直接注入 Room。
- 影响：与“单一入口管理 Room”的目标冲突，后续很难保证一致性。

3. Player ID Rule（id 不可变）
- 文档要求 id 不可改写。
- 现状：Player 仍公开 setId(...)。
- 影响：账号身份可能被覆盖，历史记录与在线状态可能错绑。

4. Player Deduplication（按 deviceId 去重）
- 文档要求加入前按 deviceId 去重。
- 现状：Player 模型内无 deviceId 字段；Room 只提供按 id/ip/name 查询。
- 影响：该规则在当前数据模型下无法实现，文档不可执行。

5. Identity Rule（禁用IP身份，改 deviceId）
- 文档要求只用 deviceId。
- 现状：DiscoveryService 周期任务仍基于 selfIp + room.findByIp(...) 更新心跳。
- 影响：身份体系和文档目标相反，且IP变化时稳定性不足。

6. UDP Socket Usage（单线程 receive）
- 文档提示多线程 receive 风险。
- 现状：代码中既有 receiverThread.receiveLoop，也在 performElection 中直接 socket.receive（同一socket）。
- 影响：存在竞争读取风险，易出现偶现丢包/状态错判。

7. Room Management（单实例 + 禁止外部 new Room）
- 文档要求 Room 由 RoomManager 统一托管。
- 现状：RoomManager.createRoom 直接 new Room()，且并非“全局唯一单房间”模型（Map多房间）。
- 影响：与文档“单房间单实例”口径冲突，术语与实现不一致。

8. Resource Handler（避免覆盖 /api /ws）
- 文档担心 /** 覆盖接口。
- 现状：WebMvcConfig 虽做了 api/ws 前缀判断，但注释写“不会进这里”，表述与实现语义存在混淆。
- 影响：维护人员容易误判路由行为；需要补充可验证用例。

三、文档本身的结构性缺陷
1) 缺“当前状态基线”
- 没说明哪些已完成、哪些未完成、对应提交号/版本号。

2) 缺“验收标准（DoD）”
- 没有可执行检查项（如：接口行为、并发测试、重复登录测试、主机切换测试）。

3) 缺“迁移计划”
- deviceId 替代 IP 涉及前端握手、后端存储、历史数据兼容，文档未给迁移步骤。

4) 缺“回滚策略”
- 涉及身份体系和发现协议，线上问题风险高，文档未给降级方案。

四、建议的修订方向（给后续“项目文档2.0”）
1) 先做“现状核对清单”
- 每条规则对应：代码位置、当前状态、目标状态、负责人、优先级。

2) 按“可落地顺序”拆分阶段
- 阶段A：身份模型补齐（先引入deviceId字段与协议透传）
- 阶段B：去IP化改造（查询/心跳/会话绑定）
- 阶段C：Room访问统一（依赖注入与职责收敛）
- 阶段D：并发与选举稳定性（socket读取与主机切换）
- 阶段E：验收与回归测试

3) 每阶段必须包含
- 改动文件清单
- 接口影响
- 数据兼容策略
- 自动化测试点
- 回滚方案

最终建议：
先修正文档为“现状+计划”格式，再启动2.0执行；否则会出现“按文档开发但无法验收”的情况。
